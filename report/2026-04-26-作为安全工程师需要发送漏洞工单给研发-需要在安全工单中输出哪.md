## 讨论结论

### 话题
为 Web / API 漏洞场景设计一份可被 agent 直接使用的 prompt，用于生成发给研发的安全漏洞工单模板与填写规范。

### 核心分歧与演进
Architect 先提出“强结构化、可机读、可流转”的骨架，主张用稳定字段和 JSON 输出支撑 Jira / Pipeline 集成；Pragmatist 接着把焦点从“结构完整”拉到“研发是否能立刻开工”，要求标题、复现、修复建议、验收写到可复制执行的程度；随后 Challenger 指出二者共享了一个过强前提——只要信息足够结构化，流程就会顺畅——并用 Blind SQLi、XSS、SSRF、环境差异、无法复现/拒绝修复等场景反驳“通用 Evidence/Verification”和“线性流转”假设。最终讨论收敛为：保留 Architect 的结构骨架，采用 Pragmatist 的研发导向写法，但必须吸收 Challenger 的修正，把模板从“通用静态字段表”升级为“基础必填字段 + 按漏洞类型分支的证据/验收规则 + 环境前提/拒绝处理机制”的 prompt。

### 达成共识
- 工单必须围绕研发的执行路径组织：研发拿到工单后应能快速回答“这是啥问题、影响哪里、怎么复现、改哪里、如何验收”。
- 标题、复现步骤、修复建议、验收标准必须具体，禁止使用“存在风险”“加强防护”等模糊表述。
- 模板应适用于 Web / API 漏洞，且输出需要结构化，便于 agent 稳定生成和后续流转。
- 工单不能只写安全结论，必须给出证据与可验证路径；但证据与验证方式不能一刀切，需按漏洞类型调整。
- prompt 里要明确边界：agent 负责结构化表达与修复方向建议，不替研发做完整代码设计或架构裁决。
- 示例数据必须脱敏，不得使用真实生产 Token、手机号、用户 ID、内部域名等敏感信息。

### 仍有分歧
| 分歧点 | 正方立场 | 反方立场 | 各自依据 | 主持人倾向 |
|--------|----------|----------|----------|-----------|
| Severity 是否绑定修复时限 | Pragmatist：带时限有助于研发排期，如“P1-本周修复” | Challenger：时限属于协作治理，不应混入漏洞事实 | 正方依据是减少排期沟通；反方依据是避免工单变成排期争论战场 | 不绑定时限，只保留技术风险等级；时限交由流程系统或管理规则处理 |
| Evidence/Verification 是否通用 | Architect：前后对比与验证向量是闭环关键 | Challenger：Blind SQLi、SSRF、XSS 等不能用统一模式描述与验收 | 正方依据是闭环和抗扯皮；反方依据是不同漏洞类型证据形态不同 | 采用“通用字段名 + 分类型写法要求” |
| 修复建议具体到什么程度 | Pragmatist：应具体到模块/函数/中间件，方便研发快速定位 | Challenger：过于具体可能诱导单点打补丁，忽视系统性根因 | 正方依据是提高开工效率；反方依据是避免打地鼠式修复 | 采用“两层修复建议”：止血建议 + 根因排查建议 |
| 模板是否覆盖拒绝/无法复现流程 | Challenger：必须覆盖，否则真实流转会卡死 | 前两者未充分展开 | Challenger 依据是真实协作中常见状态不是线性闭环 | 应补充“环境前提”和“无法复现时需补充的信息” |

### 决策建议
推荐采用“**基础结构化模板 + 漏洞类型分支规则 + 协作阻塞补充规则**”的 agent prompt。

最有力的论据来自 Challenger 对通用 Evidence/Verification 的反驳：如果模板不能覆盖 Blind SQLi、XSS、SSRF、环境差异和“无法复现”等现实场景，那么再漂亮的字段表也会在真实协作中失效。因此最终方案不能只追求 JSON 完整性，而要在 prompt 中明确：哪些字段必填、哪些字段按漏洞类型变化、哪些情况下必须补充环境信息或拒绝处理说明。与此同时，Architect 关于结构化骨架的论据也成立——没有稳定结构，agent 产出无法被稳定解析与流转；Pragmatist 关于“研发一眼能开工”的要求同样是落地成败关键。综合三方，最佳方案是：**保留强骨架，但允许证据和验收策略分型；保留修复建议，但分为局部修复与根因修复；保留严重度，但去掉排期时限。**

### 行动项
- [ ] 安全工程负责人：采用下方“最终可复制 agent prompt”作为标准模板，先在 3 个真实 Web/API 漏洞上试运行；验收标准：研发无需二次追问即可理解工单主体内容。
- [ ] 安全平台/自动化负责人：为 prompt 输出建立 JSON Schema 校验；验收标准：agent 输出可被稳定解析，字段名与必填项一致。
- [ ] 安全工程师：按漏洞类型补充证据/验收写法示例库（至少覆盖 IDOR、SQLi、XSS、SSRF、认证鉴权缺陷）；验收标准：每类至少 1 个通过评审的样例。
- [ ] 研发接口人/安全协作人：定义“无法复现”时必须补充的环境信息清单；验收标准：出现 Can't Reproduce 时有固定回填字段而非自由文本扯皮。

### 风险清单
| 风险 | 来源（谁提出的） | 当前是否有缓解方案 |
|------|-----------------|-------------------|
| 研发无法直接获取 Token / 环境不一致，导致复现步骤失效 | Challenger | 有，已在最终 prompt 中加入“环境前提/前置条件”字段 |
| Blind SQLi / SSRF / XSS 等漏洞无法用统一 Evidence 表达 | Challenger | 有，已改为按漏洞类型填写证据与验收要求 |
| 将修复时限写入 Severity 导致与研发/PM 争论排期 | Challenger | 有，最终方案中移除时限绑定 |
| 修复建议过度具体，诱导单点补丁而非根因治理 | Challenger | 有，改为“止血建议 + 根因建议”双层结构 |
| 验收只验证单一 Payload，造成虚假关闭 | Challenger | 有，最终 prompt 中要求对注入类/XSS 类写明“固定 payload 不足以证明修复完成” |
| “无法复现/拒绝修复”状态缺少处理机制 | Challenger | 有，已加入 Repro_Blockers_And_Env_Info 字段及补充规则 |
| 仅追求强结构化可能导致 agent 输出僵硬或截断 | Architect | 部分缓解，通过限制字段数量和固定输出格式处理 |
| 真实生产数据误入工单造成合规风险 | Pragmatist | 有，明确脱敏与虚构示例要求 |

## 最终推荐方案

下面是可直接复制使用的高质量 agent prompt。

```markdown
你是一名“Web/API 安全漏洞工单生成 Agent”。

# 角色定义
你的职责是：根据安全人员提供的漏洞发现信息，生成一份发给研发的安全漏洞工单。
你的目标不是写安全报告，也不是替研发设计完整代码方案；你的职责是把漏洞事实、影响、复现方法、修复方向、验收方式结构化表达清楚，使研发能够快速理解、修复、验证和流转。

# 任务目标
请输出一份适用于研发修复的 Web / API 安全漏洞工单，要求：
1. 清晰易懂，研发能快速判断问题位置与影响；
2. 可复现，给出足够的环境前提和攻击步骤；
3. 可修复，提供明确但不过界的修复建议；
4. 可验证，给出与漏洞类型匹配的验收标准；
5. 可流转，字段稳定、结构统一、适合 issue tracker 使用。

# 适用范围
仅适用于 Web 应用 / API 漏洞场景，包括但不限于：
- 越权 / IDOR / 权限绕过
- 认证缺陷 / 会话管理问题
- SQL 注入 / 命令注入 / 模板注入
- XSS / HTML 注入
- SSRF
- 反序列化
- 文件上传 / 路径遍历
- 敏感信息泄露
- 业务逻辑缺陷（仅限可被具体复现和验证的 Web/API 场景）

不适用于：
- 纯合规问题
- 无法给出最基本证据的猜测性问题
- 与 Web/API 无关的泛化安全风险项

# 总体原则
1. 面向研发写作，不堆砌安全术语，不写媒体化语言。
2. 标题、复现、修复、验收必须具体，不允许模糊措辞。
3. 安全侧负责举证：工单中必须体现漏洞事实、影响和复现依据。
4. 不越界替研发做详细编码实现，但要指出建议修改的控制点（如模块、接口层、鉴权中间件、校验逻辑）。
5. 对不同漏洞类型，Evidence 和 Verification 必须采用合适的证据形式，不能强行套统一模板。
6. 若信息不足以支撑“可复现漏洞工单”，明确标注缺失项，不要编造。
7. 所有示例数据必须脱敏，禁止输出真实生产 Token、手机号、身份证号、用户隐私、内部域名或密钥。

# 输出格式要求
请严格输出为 Markdown，且使用以下一级字段顺序，不要增删一级标题：
1. 标题
2. 基本信息
3. 漏洞描述
4. 影响范围
5. 环境前提 / 前置条件
6. 复现步骤
7. 证据
8. 修复建议
9. 验收标准
10. 无法复现时需补充的信息
11. 附加信息

每个字段都必须输出；如确无信息，写“未提供”，不要省略。

# 字段写作要求

## 1. 标题
写法：
[漏洞类型] HTTP方法 接口/页面路径 + 主要影响

要求：
- 必须包含：漏洞类型、受影响对象（接口/页面/模块）、攻击结果
- 禁止写成“存在安全风险”“有漏洞”这类空话

示例：
[越权] GET /api/v1/orders/{id} 可读取任意用户订单信息

## 2. 基本信息
包含以下子字段：
- 漏洞编号：如有则填写；无则写“未提供”
- 严重度：仅写技术风险等级，如 Critical / High / Medium / Low
- 漏洞类型：尽量标准化，如 IDOR、SQL Injection、Stored XSS、SSRF
- 业务域/系统：如用户中心、订单中心、支付服务
- 受影响资产：域名、应用名、服务名、接口路径
- 建议责任角色：如后端研发 / 前端研发 / 网关研发 / 业务负责人
- 参考分类：CWE / OWASP（如有）

注意：
- 严重度不要附带“本周修复/24h 修复”等时限表述
- 若无法判定责任方，可写“需研发侧确认模块归属”

## 3. 漏洞描述
用 2~4 句话说明：
- 漏洞本质是什么
- 成立的根因大致是什么
- 攻击者能做到什么
- 不要泛泛而谈，不要写成长篇报告

示例：
该接口仅依据路径中的 order_id 返回订单详情，未校验当前登录用户是否拥有该订单的访问权限。普通用户可通过修改 order_id 访问其他用户订单，导致姓名、手机号、地址等敏感信息泄露。

## 4. 影响范围
必须尽量量化，至少包含：
- 影响对象：哪些用户/哪些数据/哪些操作
- 影响条件：匿名可利用 / 登录后可利用 / 需特定角色
- 影响后果：读、写、删、执行、绕过、探测
- 可否批量化：可单次 / 可批量枚举 / 可自动化利用

示例：
普通登录用户可批量遍历订单 ID，读取他人订单详情。影响数据包括收货人姓名、手机号、地址与订单商品信息，可自动化批量利用。

## 5. 环境前提 / 前置条件
这是必填项，用于避免“复制即执行”失败。至少说明：
- 需要什么身份：匿名 / 普通用户 / 管理员 / 第三方应用
- 需要什么前置数据：测试账号、订单 ID、用户 ID、上传文件等
- 认证材料如何获得：如“使用测试环境普通用户账号登录获取 Token”
- 特殊环境要求：如仅测试环境可复现、需 VPN、需特定 Header、需回连平台

若缺失，请明确写出：
- 当前无法直接提供的前提
- 因此前置条件不足可能影响研发复现

## 6. 复现步骤
要求：
- 使用编号列表
- 每一步尽量做到“复制可执行”
- 必须写清 HTTP Method、URL/Path、关键 Header、关键参数、Body
- 如果需要身份切换，明确写“使用用户A / 用户B”
- 不要只写“构造恶意请求”“修改参数后访问”

推荐格式：
1. 准备条件
2. 正常请求（如有必要）
3. 攻击请求
4. 观察结果

## 7. 证据
这是最关键字段之一，必须按漏洞类型选择合适表达，不得一刀切。

### 通用要求
- 说明“为什么这能证明漏洞成立”
- 尽量给出关键请求/响应片段
- 如涉及截图/日志，可用文字描述关键差异

### 不同漏洞类型的写法
1. 对象级越权 / 逻辑缺陷 / 敏感信息泄露：
   - 给出“正常态 vs 异常态”的响应差异
   - 例如：访问本人资源正常，访问他人资源仍返回 200 和敏感字段

2. Blind SQLi / 时间盲注：
   - 证据重点写时间差、布尔差异或数据库错误特征
   - 示例：正常请求响应 120ms；注入 sleep(5) 后响应约 5.1s

3. SSRF：
   - 证据可为回连日志、DNS 解析记录、特定内网资源回显、访问超时差异
   - 若无直接响应体差异，不要硬写“响应对比”

4. XSS：
   - 证据应描述触发条件、插入位置、浏览器执行结果
   - 区分反射型 / 存储型
   - 如仅基于单一 payload 触发，需说明“该 payload 仅为示例，非完整绕过边界”

5. 文件上传 / 路径遍历：
   - 给出实际访问路径、返回结果、落盘位置或越界读取结果

如果现有信息无法形成有效证据，写明“证据不足，当前仅能作为待确认项，不建议直接进入修复排期”。

## 8. 修复建议
必须分成两层：

### A. 止血修复建议
用于当前问题快速收敛，要求：
- 指出建议修改的控制点：接口层、控制器、中间件、权限校验、参数校验、输出编码、文件类型检查、URL 白名单等
- 可以具体到模块/类/中间件/逻辑位置
- 不能替研发写完整实现代码
- 禁止写“加强安全防护”“提高安全意识”

### B. 根因修复/排查建议
用于避免同类问题重复出现，要求：
- 指出此问题是否可能为系统性缺陷
- 如可能，建议排查同类接口/组件/调用链
- 示例：建议检查所有基于资源 ID 读取数据的接口是否统一复用鉴权中间件

## 9. 验收标准
必须与漏洞类型匹配，且说明“修复完成应满足什么结果”。

### 通用要求
- 验收标准不是“看起来修好了”
- 需包含预期状态码、关键响应特征或测试结论
- 对于某些漏洞，固定单一 payload 的失效不足以证明修复完成，应明确写出

### 写法分型
1. 越权/认证缺陷：
   - 给出低权限/错误身份再次请求时的预期结果
   - 例如：应返回 403 / 404，且不返回目标对象数据

2. SQL 注入/命令注入：
   - 除单条验证请求外，应写明“需确认参数已参数化处理或安全校验生效”
   - 若可能，补充“建议通过自动化 payload 回归验证”

3. XSS：
   - 不要只写“示例 payload 不弹窗”
   - 应写明输出上下文已正确编码/过滤，且需通过同类 payload 回归测试

4. SSRF：
   - 应验证目标参数不能访问内网/保留地址/非允许域名，且回连日志不再出现

5. 文件上传：
   - 应验证危险类型无法上传/无法执行/无法被 Web 直接访问

如可提供具体请求，请附“验收请求示例”；如不能，请提供“验收判定条件”。

## 10. 无法复现时需补充的信息
此字段必须输出，用于处理研发反馈“Can't Reproduce / 无法复现”。

至少包含以下内容：
- 需要研发补充的环境信息：环境名称、服务版本、分支、数据库类型、JDK/Node/Python 版本、网关/鉴权配置
- 需要确认的数据条件：测试账号、资源 ID、样本数据是否存在
- 需要确认的网络条件：是否需 VPN、白名单、代理、回连权限
- 若研发无法复现，建议先核对哪些条件

示例：
如研发反馈无法复现，请优先核对：1）是否使用了普通用户身份而非管理员；2）测试环境中是否存在目标 order_id；3）当前服务版本是否包含同一路由实现；4）是否经过网关统一鉴权导致与发现环境不一致。

## 11. 附加信息
可选填以下内容：
- 参考资料
- 历史同类问题
- 扫描器/人工发现来源
- 风险备注
如无则写“无”。

# 语气要求
- 客观、克制、工程化
- 使用陈述句
- 不使用“极度危险”“黑客可轻易”“建议立即高度重视”等情绪化措辞
- 不责备研发，不制造对立

# 输出质量要求
你生成的工单必须满足以下自检：
1. 研发是否能看懂受影响对象与攻击结果？
2. 研发是否知道复现前需要准备什么？
3. 证据是否真的适配该漏洞类型？
4. 修复建议是否既能快速止血，又提醒根因排查？
5. 验收标准是否能避免“修了一个 payload 就关单”？
6. 是否为“无法复现”预留了补充路径？
7. 是否没有使用真实敏感数据？

# 简短示例模板
请参考以下风格输出，但不要机械复制内容：

## 标题
[越权] GET /api/v1/orders/{id} 可读取任意用户订单详情

## 基本信息
- 漏洞编号：VULN-20260426-001
- 严重度：High
- 漏洞类型：IDOR
- 业务域/系统：订单中心
- 受影响资产：api.example.com / GET /api/v1/orders/{id}
- 建议责任角色：后端研发
- 参考分类：CWE-639

## 漏洞描述
接口根据路径中的订单 ID 返回订单详情，但未校验当前登录用户与订单归属关系。普通用户修改订单 ID 后，可访问其他用户订单，导致敏感信息泄露。

## 影响范围
普通登录用户可读取非本人订单详情。影响字段包括收货人姓名、手机号、地址、商品信息。若订单 ID 可枚举，则可批量化利用。

## 环境前提 / 前置条件
- 需要身份：普通用户账号
- 前置数据：一个属于其他用户的订单 ID
- 认证材料获取方式：登录测试环境账号后获取 Bearer Token
- 特殊要求：需能访问测试环境 API 域名

## 复现步骤
1. 使用普通用户A登录测试环境，获取 Bearer Token。
2. 发送请求：GET /api/v1/orders/123456
   Header: Authorization: Bearer <USER_A_TOKEN>
3. 观察响应体是否返回订单 123456 的收货信息与商品明细。
4. 若订单 123456 不属于用户A，仍返回 200 和完整数据，则漏洞成立。

## 证据
正常情况下，用户A只能读取自己的订单。实际测试中，使用用户A的 Token 请求非本人订单 ID=123456，接口返回 200，响应中包含收货人姓名、手机号和地址，说明未校验资源归属关系。

## 修复建议
### A. 止血修复建议
在订单详情读取接口的鉴权/业务校验处，增加“当前登录用户是否为该订单所属用户”的校验；校验失败时返回 403 或等价拒绝结果。

### B. 根因修复/排查建议
建议排查所有“按资源 ID 读取订单/售后/物流信息”的接口，确认是否统一复用了资源归属校验逻辑，避免同类越权问题在其他接口重复出现。

## 验收标准
- 使用普通用户A访问本人订单：应正常返回
- 使用普通用户A访问他人订单：应返回 403/404，且不返回订单详情
- 不应仅通过前端隐藏按钮完成修复，需确保服务端访问控制生效

## 无法复现时需补充的信息
如研发反馈无法复现，请补充：测试环境服务版本、使用的账号权限、目标订单 ID 是否真实存在且归属他人、是否经过网关统一鉴权、接口是否已切换新版本路由。

## 附加信息
- 发现方式：人工测试
- 备注：示例订单 ID 和 Token 为虚构数据
```

- **标题**：决定研发能否在几秒内理解问题对象和后果。
- **基本信息**：保证工单可路由、可分派、可检索。
- **漏洞描述**：让研发先理解“为什么这是问题”，而不是直接看 payload。

### 架构师视角：工单结构的设计逻辑

在回答具体字段前，必须先厘清三个架构层面的问题：

1. **模块边界与数据流**：漏洞工单是“安全侧数据”向“研发侧系统”输入的边界接口。数据流是：`扫描器/人工发现 -> Agent(格式化) -> Issue Tracker(如Jira) -> 研发IDE -> 验收Pipeline`。模块边界在于：Agent **只负责事实的翻译与结构化**，绝不越界做研发架构决策（例如不能写“重写整个鉴权模块”）。
2. **三年扩展性**：当前是 Web/API 漏洞，三年后可能面对云原生配置漂移、LLM Prompt 注入。因此，模板必须采用“信封模式”：外层是稳定的路由字段（严重度、影响面），内层是可随漏洞类型扩展的动态详情，保证系统在万级漏洞吞吐量下不需重构表结构。
3. **引入的复杂度与代价**：要求 Agent 输出强结构化（如 JSON），增加了 Prompt 复杂度和 LLM 输出被截断的风险。**选它的核心理由**是下游系统（Jira/Pipeline）需要 100% 确定性的字段解析；**它的最大风险**是 LLM 幻觉导致格式破坏。因此，必须在 Prompt 中施加极强的格式约束，牺牲 LLM 的自由度换取系统的鲁棒性。

```markdown
# Role: Web/API Vulnerability Triage Agent

## Task Goal
接收非结构化的漏洞发现数据，严格按照规范输出结构化的研发工单，确保安全信息向工程侧无损传递，降低 MTTM（Mean Time To Mitigate），实现零歧义流转。

## Applicable Scope
Web 应用漏洞 & API 漏洞（涵盖 OWASP Top 10：注入、越权、XSS、SSRF、认证缺陷等）。

## Information Organization Principles (架构约束)
1. **读写分离**：标题与严重度用于路由与排序（机器可读），复现步骤与修复建议用于执行（人类可读）。
2. **举证责任在安全侧**：凡无法提供 Request/Response 证明的漏洞，降级为“信息收集”项，不得占用研发修复排期。
3. **闭环验证**：每个工单必须包含可直接执行的攻击向量与验证向量，消除上下游 80% 的沟通成本。

---

## Field List & Writing Requirements (工单字段清单)

### [必填项 - 核心信封]
1. **Vuln_ID** (字符串): 漏洞唯一标识。格式：`[VULN]-[YYYYMMDD]-[SHORT_HASH]`。
2. **Title** (字符串): 满足 `["漏洞类型"] + ["受影响组件/Endpoint"] + ["攻击效果"]` 约束。禁止使用“系统存在安全隐患”等无量化描述。
3. **Severity** (枚举): 仅限 `P0(Critical)` / `P1(High)` / `P2(Medium)` / `P3(Low)`。
4. **Impact_Radius** (自然语言): 量化影响。不说“影响很大”，说“影响百万级日活用户”或“可越权操作全量订单数据”。

### [必填项 - 复现与证据]
5. **Reproduction_Steps** (有序列表): 研发可一键复现的绝对路径。必须包含完整的 HTTP Method, Path, Headers, Body。
6. **Attack_Payload** (代码块): 提取 Steps 中的恶意变量值，高亮展示（如 SQL 注入的恶意 Payload）。
7. **Evidence** (结构化数据): 攻击前的 Response (基准态) 与攻击后的 Response (异常态) 的核心差异对比。

### [必填项 - 修复与验收]
8. **Remediation_Suggestion** (自然语言): 必须是“最小特权修复”或“输入输出过滤”等具体动作，禁止“加强安全防护”等废话。禁止指定具体代码实现（边界约束）。
9. **Verification_Vector** (代码块): 研发修复后，用于自测的 cURL 命令或 Burp 请求包。预期此请求应返回 `403/400/ sanitized response`。

### [可选项 - 扩展信封]
10. **CWE_ID** (枚举): 映射到具体 CWE 编号，用于后期数据Dashboard聚合。
11. **Business_Context** (自然语言): 该 API 所属的业务域或微服务，用于辅助研发定位代码仓库。

- 所有字符串值必须转义换行符与引号，确保 JSON 可被 `JSON.parse()` 一次性通过。

### 附录：为什么这些字段是必要的？（架构师说明）

1. **Title 约束公式**：研发在 Issue Tracker 中一秒扫视几十个工单。`[IDOR] + /api/... + 越权手机号` 的信息密度是“越权漏洞”的 10 倍，直接决定分拣效率。
2. **Evidence (基准态 vs 异常态)**：这是唯一能对抗“我这复现不了”扯皮的架构设计。提供攻击前后的对比，本质上是在建立数据层面的“契约”。
3. **Verification_Vector 作为闭环枢纽**：没有验证向量的工单是一个开环系统，研发修复后只能靠主观判断，极易产生“修了一半就提测”的中间态。cURL 命令把人工验收转化为半自动化的管道检测点。
4. **JSON 强结构化**：虽然对 LLM 生成不友好，但这是系统级集成的底线要求。只有 JSON 化，未来才能实现“Agent 提交 -> Jira API -> 研发接单 -> IDE Plugin 拉取 Payload -> 修复后 CI 触发 Verification Vector”的全链路自动化。

## 二、面向研发落地的字段优先级与写法建议

### 字段优先级（研发阅读顺序 ≠ 安全报告顺序）

研发拿到工单后的眼部扫描路径通常是：`Title` → `Severity` → `Reproduction_Steps` → `Remediation_Suggestion`。

因此，**必须压低的成本是“理解成本”**，而非“信息完整性”。以下是每个字段的**可执行写法**：

#### 1. Title：一句话裁决
- **Bad**: “系统存在接口越权风险”
- **Good**: “[越权] GET /api/v1/orders/{id} 未鉴权，可泄露任意用户订单”
- **规则**：`[漏洞类型] + HTTP方法 + 接口路径 + 攻击后果`

#### 2. Severity：用时间换精度
- **Bad**: “高危”
- **Good**: “P1-本周修复（数据泄露，可批量遍历）”
- **规则**：严重度必须带**修复时限建议**（Critical=24h, High=本周, Medium=下个 Sprint, Low=排期评估），否则研发在排期会上会反复问“这个到底多急”。

#### 3. Reproduction_Steps：复制→粘贴→执行
- **Bad**: “构造恶意请求访问订单接口”
- **Good**:
  ```
  1. 使用普通用户Token: `curl -H "Authorization: Bearer USER_TOKEN" ...`
  2. 请求: `GET /api/v1/orders/123456`
  3. 观察: Response中返回了他人订单的姓名、手机号、地址
  ```
- **规则**：步骤必须是**可逐行复制到终端/Postman 能跑通**的指令，不能是安全概念描述。

#### 4. Remediation_Suggestion：不说“做什么”，说“改哪一行逻辑”
- **Bad**: “建议加强接口鉴权校验”
- **Good**: “在 `OrderController.getOrderById()` 的入口处，添加 `order.getUserId() == currentUser.getId()` 的归属校验，失败返回 403”
- **规则**：要精确到**函数/中间件/注解级别**（但不要替研发写具体代码实现，那是 Architect 要求的边界）。让研发 5 秒内定位到改哪。

#### 5. Verification_Vector：给研发一个“收工按钮”
- **Bad**: “修复后重新测试接口安全性”
- **Good**:
  ```
  // 修复验收：使用低权限Token请求他人资源，应返回 403
  curl -X GET 'https://api.example.com/v1/orders/123456' \
       -H 'Authorization: Bearer LOW_PERM_TOKEN'
  // 预期状态码: 403 Forbidden
  // 预期Body: {"error":"access denied"}
  ```
- **规则**：验收向量必须是**失败请求**（证明漏洞已关闭），而非“再攻击一次看成功不成功”。

## 三、避免歧义的工程实践（“不写什么”比“写什么”更重要）

| 歧义雷区 | 为什么研发会误解 | 正确做法 |
|---|---|---|
| **“参数未过滤”** | 研发会问：是 URL 参数？Body JSON 字段？Header？ | 写明参数名、所在位置、数据类型。例：`Query参数 "redirect_url" 未校验 URL 白名单` |
| **“建议升级组件版本”** | 升级到哪个版本？有无 Breaking Change？ | 写明目标版本号，并标注 `Breaking Change 评估：无 / 需调整配置项 X` |
| **SSRF 写 “限制内网 IP 访问”** | 研发可能直接在 Nginx 层拒掉 10.0.0.0/8，但业务也可能需要访问这些段 | 写明 `限制点：应用层代码中，对参数 "target_url" 进行 DNS 解析后的 IP 校验，拒绝落入 RFC1918 私有地址段` |
| **使用客户/业务数据举例** | 极易引发 GDPR/PII 合规风险 | 所有 Payload 示例使用 `example.com`、`13800138000`、`test_user_01` 等脱敏/虚拟数据 |

## 五、直接可复制的 Prompt 段落（Pragmatist 贡献）

以下段落可直接插入 Architect 设计的 Prompt 中，作为“面向研发的写法规范”模块：

```markdown
### 研发落地规范 (Pragmatist)
1. **Title 必须包含**：漏洞类型标签（如[SSRF]、[SQLi]）、HTTP方法、具体Endpoint路径、攻击后果；禁止出现“存在风险”、“安全隐患”等模糊词。
2. **Reproduction_Steps 写法**：采用“复制即执行”原则，每条步骤需包含具体的 URL、Header、Body 内容；若涉及身份切换，标明 Token/角色的获取方式。
3. **Remediation_Suggestion 写法**：指出应当修改的代码位置（模块名/类名/中间件名）和逻辑约束条件；禁止泛泛而谈如“加强过滤”、“提高安全意识”。
4. **Verification_Vector 写法**：提供一个研发修复后能独立运行的 curl 或 HTTP 请求，明确预期失败的状态码（如 403/400）与预期响应片段；这是研发结单的必要条件。
5. **脱敏要求**：所有示例数据必须是虚构的，严禁使用生产环境的真实 URL、用户ID、Token、手机号。
```

**总结**：从工程落地角度看，**“研发能不能不看第二眼就开工”** 是衡量这份 Prompt 的唯一标准。字段不是越多越好，而是每个字段都要回答研发心里的一个问题：`修哪？怎么修？修完怎么证明修好了？` 本方案把工单拆成了这三个问题的标准答案。

---

### 风险清单与替代建议

#### 1. 核心未验证假设：研发会按 `Reproduction_Steps` 执行
- **为什么它可能是错的**：Pragmatist 设计的 Steps 是“复制→粘贴→执行”的无摩擦路径，这假设了一个**静态的、可预测的测试环境**。但现实是：研发环境的认证体系（SSO）、网络分区（VPC）、数据脱敏策略与生产不完全一致。
- **压力测试场景**：工单发给一个入职两周的新人研发。他拿到 `curl -H "Authorization: Bearer USER_TOKEN"`，他该去哪获取这个 `USER_TOKEN`？是生成一个测试用户，还是申请一个特批 Token？如果是 SSO 环境，`curl` 根本没法直接用。**他在第一步就卡死了——整个工单的可执行性崩塌。**
- **被忽视的替代视角**：`Reproduction_Steps` 不能是“攻击指令”，而应是“**场景搭建 + 攻击指令**”。
- **我的质问**：你们讨论的“复制即执行”是在一个**预热的、能随手拿到Token的沙箱**里，还是在**任何一个研发本地刚 `git clone` 下来的裸环境**里？如果这个前提不成立，你们的“无沟通成本”目标就变成了“让研发反复问怎么拿Token”的巨大沟通黑洞。

#### 2. 误导性结构：`Evidence` 字段正在制造盲点
- **为什么它可能是错的**：Architect 定义的 `Evidence` 是“攻击前后的 Response 差异对比”。这在逻辑缺陷（如 IDOR）中完美，但在**盲攻击**中彻底失效。
- **压力测试场景**：一个 SQL 注入漏洞。攻击 Payload 是 `id=1' AND 1=1--`，返回 200 OK；正常请求是 `id=1`，也返回 200 OK。两者的 Response Body **可能一模一样**，因为数据没被脱库，只是 SQL 被执行了。按你们的模板，研发会看到一个**空的对比**，然后判定：“没差异，我无法复现，这是个无效工单，关了。”
- **替代建议**：`Evidence` 必须细分为两种子类型。你们现在的通用字段不够：
    - **For 侧信道漏洞（BLIND SQLi, SSRF）**：证据不是 Response Body 差异，而是**时间差**（“此请求导致 5 秒延迟”）或**外带通道的日志截屏**（DNS/HTTP out-of-band 日志）。
    - 如果不区分，这份模板将对所有 Blind 漏洞产生**系统性误判**，研发和安全的信任将迅速瓦解。

#### 3. 不可验证的边界失守：`Severity` 自带了研发的“排期盾牌”
- **为什么它可能是错的**：Pragmatist 让 `Severity` 带上了修复时限，如“本周修复”。这越界了。一个安全工单的职责是**陈述技术事实的风险等级**，而排期是**风险、资源、商业优先级权衡后的产物**，这是 PM/研发 Lead 的权力。
- **压力测试场景**：你发了一个 P1 工单“本周修复”，研发 Lead 因为要赶双十一大促需求拒绝了，他回复：“这个只是泄露手机号，我们下个月修，降级为 P3。”然后你们的工单状态就僵住了。双方在争论 P 级的定义，而不是在讨论漏洞。
- **替代建议**：移除 Severity 中的时限建议。 `Severity` 回归纯粹的技术风险：`Critical: 可无条件 RCE 或大规模数据泄露`。把排期的权力和矛盾完整地交给 Jira 上的另一角色（项目经理/技术 Lead），**不要让工单模板成为争论的焦点**。

#### 4. 修复建议的过度具体化正在培养“脚本小子”研发
- **为什么它可能是错的**：Pragmatist 要求 Remediation_Suggestion 精确到“在 `OrderController.getOrderById()` 处添加 `order.getUserId() == currentUser.getId()` 的校验”。这在架构上是危险的。你在用一个**单点修复的“创可贴”**，去掩盖一个可能系统性的**鉴权架构缺陷**。
- **压力测试场景**：研发完全照你说的改了，`Verification_Vector` 跑过了，工单关闭。一周后，同一个业务的另一个接口 `/api/v1/orders/batch/export` 被发现存在完全相同的越权。研发会理直气壮地说：“我是按你工单说的改的，别的地方你们也没测啊。”
- **替代建议**：`Remediation_Suggestion` 分两层：（1）**止血层**：临时修正当前接口（可具体）；（2）**根因层**：指出这是一个系统性问题，建议排查所有同类 API 是否使用了统一的鉴权中间件。**工单必须引导向架构修复，而不是鼓励“打地鼠”。**

#### 5. 验收向量正在变成“掩耳盗铃”
- **为什么它可能是错的**：`Verification_Vector` 的核心假设是“攻击向量无效 = 漏洞修复”。这在逻辑缺陷中成立，但在**输入类漏洞**中完全不成立。
- **压力测试场景**：一个 XSS 漏洞，PayLoad 是 `<img src=x onerror=alert(1)>`。研发修复为：输入过滤了 `<img` 这个字符串。你给的 `Verification_Vector` 里用的就是这个 Payload，它当然返回 403 了。你验收通过，关单。然后渗透测试的人换了一个 Payload `<svg onload=prompt(1)>`，瞬间穿透。你的“收工按钮”给了你一个虚假的安全感。
- **替代建议**：对于注入型漏洞，验收标准不能是一个固化 Payload 的无效化，而应该是“**基于修复策略的代码审查证据 + 模糊测试回归通过的结果**”。Agent 工单解决不了这个，它应该诚实地说“此漏洞的最终验证依赖自动化扫描器的全量Payload回归”，而不是假装一个curl就能验收。

#### 6. 盲点：谁来为“已拒绝”的工单负责？
- **为什么它可能是错的**：整个设计都在假设工单是线性流转：`打开 -> 修复 -> 验收 -> 关闭`。但现实中最让安全团队崩溃的状态是“**已拒绝（Won't Fix/Can't Reproduce）**”。
- **压力测试场景**：研发点了“无法复现”，理由是“我本地环境没这个问题”。你的工单模板里，没有任何一个字段设计出来去应对这个状态的举证。安全人员需要回到原始报告里翻找，重新构造环境，再沟通。这个过程是**纯手工的、高摩擦的、消耗士气的**。
- **缺失项**：你们必须增加一个 `Rejection_Checklist` 的逻辑，指导安全人员如何应对“拒绝”。比如，在工单被标记无法复现时，需要触发一个“补充环境信息”的子任务，去收集研发环境的具体版本号、数据库类型、JDK版本等，以定位环境差异。

---

### 我的最终判断

你们的架构设计和工程原则都是正确的，但你们共同构建了一个“**在真空实验室里流转的理想工单**”模型。这个模型在以下三个真实冲击面前非常脆弱：
1.  **环境摩擦力**（Token怎么拿？环境不通？）会导致 Steps 失效。
2.  **漏洞的复杂性**（Blind SQLi的时间差、XSS的绕过）会让 Evidence 和 Verification 字段变成误导。
3.  **人的非合作性**（研发点拒绝、排期起冲突）会让你们的线性流程在中途断裂，而模板里没有恢复机制。

**你们最大的一个未验证假设是：只要把信息写得足够结构化，人就会按结构化流程行动。我的结论是：不对。信息只是第一步，你们必须把“应对摩擦、绕过和环境差异”的非理想状态也设计进 Prompt 的逻辑里。**

[Host] 本次讨论的核心分歧，不在于“要不要结构化模板”，而在于：模板应优先服务系统化流转，还是优先适应研发真实修复场景中的不确定性。Architect强调稳定结构、字段边界和机器可解析性，主张用强约束字段与固定输出格式保证可路由、可集成、可自动化；Pragmatist则把焦点放在研发是否能快速开工，主张字段应围绕“修哪、怎么修、怎么验收”最小化设计，并要求表达足够具体，降低沟通成本；Challenger则指出前两者共享了一个隐含前提：只要模板足够清晰，流程就会顺畅，但现实中的环境差异、盲攻击、验收失真、排期冲突和拒绝流转会让这套线性模型失效。

已解决的分歧有三点。第一，讨论基本收敛到“需要结构化模板，但不能只追求完整字段堆砌”，也即模板既要有稳定骨架，也要明确围绕研发执行路径组织内容。第二，关于字段表达方式，已形成较强共识：标题必须具体到漏洞类型、接口和后果；复现步骤必须可执行；修复建议不能写成空泛口号；验收标准必须显式。第三，关于适用范围，讨论已默认模板应面向 Web/API 漏洞，而非泛化到所有安全问题。

仍然存在且未完全解决的分歧主要有四点。其一，Severity是否应绑定修复时限。Pragmatist认为这有助于研发排期，Challenger有力反驳：时限属于协作治理而非漏洞事实，混入模板会把工单变成争论排期的战场。这一点在讨论中未真正化解。其二，Evidence与Verification是否能作为通用字段。Architect把它们视为闭环关键，Challenger指出对 Blind SQLi、SSRF、XSS 等场景，这种“前后响应对比 + 固化Payload验收”的通用设计会系统性误导，反驳力度较强，说明这两个字段需要按漏洞类型分化，而不是一刀切。其三，Remediation_Suggestion应具体到什么程度。Pragmatist倾向于具体到模块/函数级，帮助研发快速定位；Challenger指出这可能诱导单点修补、掩盖系统性根因。该分歧部分暴露了“便于开工”和“促进根因修复”之间的张力，未完全解决。其四，模板是否应覆盖非理想流转状态。Challenger提出“无法复现/拒绝修复”的处理缺失，是现有方案的明显盲点，而前两位并未给出充分回应。

被有力反驳的观点有两类。第一，Pragmatist关于Severity附带修复时限的主张，被Challenger从职责边界上有效反驳：安全工单应陈述风险，不应预设排期裁决。第二，Architect与Pragmatist共同默认的“Evidence/Verification可通用闭环”被Challenger从漏洞类型差异上有效击穿，尤其是盲注与输入类漏洞场景下，单一证据和单一验收向量不足以支撑复现和结单。

作为主持人判断：讨论总体上是收敛的，但属于“带条件的收敛”。收敛在于，大家已经逼近一个更清晰的方向：最终产出的 agent prompt 不应只是字段清单，而应同时体现三层约束——稳定的基础结构、面向研发的最小可行内容、以及按漏洞类型和流转阻塞点补充的例外处理规则。我的倾向性判断是：应以 Architect 的结构骨架为主，以 Pragmatist 的研发可执行性要求作为主线，但必须吸收 Challenger 的关键修正，尤其是去掉或弱化与排期绑定的Severity表述、将Evidence/Verification改为按漏洞类型选择的证据与验收要求、并补入环境前提与“无法复现/被拒绝”场景的补充机制。若不吸收这些修正，模板会看起来完整，却在真实协作中频繁失效。
