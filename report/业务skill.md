# 二、建议的总体架构

我建议做成一个**分层、规则可插拔、分析能力复用**的架构，而不是每种风险写一套独立逻辑。

---

## 1. 分层架构图思维

可以抽象成 5 层：

```text
[代码输入层]
  ├─ 源码 / AST / CFG / Call Graph / IR
  └─ 框架元信息（Controller、RPC、ORM、注解、路由）

[程序分析基础层]
  ├─ 参数识别
  ├─ 调用图构建
  ├─ 数据流/污点传播
  ├─ 控制流/条件依赖分析
  └─ 路径摘要生成

[业务语义抽取层]
  ├─ 敏感操作识别
  ├─ 校验动作识别
  ├─ 风控/认证/鉴权接口识别
  ├─ 状态检查识别
  └─ 业务前置条件模型化

[规则引擎层]
  ├─ 缺失校验规则
  ├─ 必要接口未调用规则
  ├─ 校验可绕过规则
  ├─ 条件弱约束规则
  └─ 多阶段流程一致性规则

[结果输出层]
  ├─ 风险点
  ├─ 证据链
  ├─ 路径说明
  ├─ 风险等级/置信度
  └─ 修复建议模板
```

---

# 三、为什么要这么分层

这是架构上最关键的点。

## 选这种分层的理由
- **参数分析、数据流、分支分析是公共能力**，不能每条规则重复实现
- “指定接口识别可扩展”天然适合放在**业务语义层**
- “是否会被绕过”本质要结合**控制流 + 数据流 + 业务语义**
- 后续如果你要扩到 3-5 倍规则规模，必须靠可复用分析中间结果，而不是 rule-by-rule 硬编码

## 不这么做的代价
如果你直接把 skill 写成：
- 一个规则匹配参数
- 一个规则看接口
- 一个规则判断 if 分支
- 一个规则看绕过

最后会出现几个问题：
1. **规则互相复制 AST 遍历逻辑**
2. **同一份调用链被反复分析，性能差**
3. **无法统一证据链**
4. **新增风险类型时，只能继续堆 if/else**
5. **绕过类检测会非常脆，误报高**

# 四、核心对象模型：你需要抽象哪些“业务元素”

这是整个 skill 能不能扩展的关键。

我建议定义以下几个统一对象：

---

## 1. EntryPoint（入口点）
代表业务请求入口，比如：

- HTTP Controller 方法
- RPC 接口
- MQ 消费处理方法
- 定时任务入口

用途：
- 确定分析边界
- 建立“一个业务动作”的起点

---

## 2. SensitiveAction（敏感操作）
代表真正需要被保护的业务动作，比如：

- 发验证码
- 修改手机号
- 重置密码
- 提现
- 下单支付
- 绑定银行卡
- 发券
- 审批通过
- 提交关键表单
- 调用下游核心服务

识别方式：
- 指定接口
- 指定方法名模式
- 注解/SDK
- 关键 DB 写操作
- 关键状态变更

---

## 3. Guard / Check（前置校验）
代表保护性动作，比如：

- 登录校验
- 权限校验
- 验证码校验
- 人机校验
- 风控校验
- 频控
- 状态校验
- 幂等校验
- token/签名校验

---

## 4. Condition（控制条件）
代表影响流程是否进入敏感操作的条件：

- `if user.isAdmin`
- `if verifyCodeValid`
- `if skipCheck == true`
- `if request.source == internal`
- `if scene in whitelist`

这个对象要记录：
- 条件表达式
- 条件依赖变量
- 变量来源

- 条件控制的代码块

---

## 5. EvidencePath（证据路径）
输出时必须给到：
- 请求参数从哪里进来
- 经过哪些函数
- 哪个分支导致绕过
- 哪个敏感操作最终被执行
- 为什么认为缺少校验/校验可绕过

这个对象是结果可解释性的核心。

---

# 五、推荐的数据流

你这个 skill 最好按下面的数据流工作：

```text
代码输入
  ↓
构建 AST / CFG / CallGraph / 基础符号表
  ↓
识别入口点 EntryPoint
  ↓
从入口点出发做：
  - 参数分析
  - 调用链展开
  - 条件依赖收集
  - 关键调用识别
  ↓
抽取业务语义：
  - 敏感操作
  - 校验动作
  - 状态检查
  - 风控/认证接口
  ↓
做路径级别比对：
  - 敏感操作前是否存在所需 Guard
  - Guard 是否控制所有到达路径
  - Guard 是否和用户输入存在弱绑定/可绕过关系
  - 是否缺失指定关键接口调用
  ↓
规则命中
  ↓
输出证据链 + 风险等级
```

# 六、你这 5 个能力，应该如何拆成模块

---

## 模块 1：参数分析器
职责：
- 识别入口参数
- 判断参数是否用户可控
- 参数是否影响关键分支
- 参数是否进入关键接口

输入：
- AST / 符号表 / 方法签名

输出：
- 参数定义表
- 可控参数集合
- 参数到变量映射关系

### 选它的理由
参数是所有业务逻辑分析的起点，尤其“绕过”很多时候就是因为某个参数参与了条件判断。

### 不选它的代价
你后续无法判断：
- 分支是否受用户输入控制
- 校验是否被请求参数关闭
- 风控条件是否可被客户端篡改

---

## 模块 2：数据流追踪器
职责：
- 从请求参数到关键调用做传播分析
- 跟踪变量赋值、包装、传参、返回值
- 支持跨函数传播

输出：
- Source → Sink 的传播路径
- 变量别名关系
- 函数间数据依赖

### 选它的理由
你要判断：
- 参数是否进入敏感接口
- 校验结果是否真正影响业务动作
- 某个 token/验证码是否被实际使用

### 不选它的代价
你只能做表面匹配，比如“看起来调用了 verify()”，但不知道 verify 结果有没有真正控制后续逻辑。

---

## 模块 3：关键接口识别器
职责：
- 判断是否调用了指定接口
- 支持可扩展配置
- 支持接口分类：鉴权、风控、发送验证、状态更新、落库等

建议做成**规则注册中心**，而不是写死在代码里。

配置可以类似：

```yaml
api_categories:
  auth_check:
    - com.xxx.auth.AuthService.checkPermission
    - com.xxx.auth.AuthFacade.verify
  risk_check:
    - com.xxx.risk.RiskService.check
  sms_send:
    - com.xxx.sms.SmsService.sendCode
  captcha_verify:
    - com.xxx.captcha.CaptchaService.verify
  state_update:
    - com.xxx.order.OrderService.updateStatus
```

### 选它的理由
这是你“需要可扩展”的核心点。新增一个业务系统时，不应该改分析框架代码，只应该补配置/规则。

### 不选它的代价
每增加一个接口族都要改代码，规则和分析引擎耦合，会很快失控。

---

## 模块 4：条件/分支分析器
职责：
- 抽取 if/switch/三元表达式/异常分支
- 建立“进入某个敏感操作需要满足哪些条件”
- 判断是否存在某条路径绕过校验

输出：
- Path Condition
- 条件变量依赖图
- 守卫条件覆盖率

### 选它的理由
绕过问题几乎都离不开它。  
不是“有没有校验”，而是“是不是所有路径都必须经过校验”。

### 不选它的代价
你会产生大量误报和漏报：
- 调用了 verify 但在旁路分支里没用
- 正常分支有校验，异常/兼容/白名单分支没校验
- 某些 return/continue/try-catch 分支导致保护失效

---

## 模块 5：绕过模式分析器
职责：
把常见业务绕过抽象成通用模式，而不是一个漏洞一个规则。

建议先抽象 5 类通用模式：

### 模式 A：缺失前置校验
敏感操作前根本没有 guard

例如：
- 修改手机号前没有验证码校验
- 提现前没有支付密码校验

---

### 模式 B：校验存在但结果未绑定
调用了校验接口，但结果没用于控制敏感操作

例如：
- `verifyCode()` 调了，但返回值没判断
- `riskCheck()` 结果被日志打印，但不影响执行

---

### 模式 C：条件可控导致绕过
是否执行校验受用户输入或弱信任条件控制

例如：
- `if (skipVerify)` 跳过验证
- `if (source == "internal")` 直接放行，而 source 来自请求参数
- `if (isTestUser(phone))` 绕过，且 phone 可控

---

### 模式 D：非全路径覆盖
部分路径有校验，部分路径没有

例如：
- 新增接口有校验，编辑接口没有
- 正常流程有验证码，异常恢复流程没有
- 一个 `if/else` 只有一边校验

---

### 模式 E：校验与动作不一致
校验对象、场景、资源与实际敏感动作不匹配

例如：
- 校验的是 A 用户，操作的是 B 用户
- 校验的是登录验证码，执行的是支付操作
- 校验针对 `mobile`，实际更新的是 `newMobile`

### 选它的理由
这些模式具有高度可复用性，可以覆盖你提到的“发送验证绕过、校验绕过等逻辑基本一致”的场景。

### 不选它的代价
你会陷入“每个业务写一条特殊规则”，难以维护。

# 七、规则引擎怎么设计才可扩展

我建议你不要只做“代码匹配规则”，而要做**两阶段规则体系**：

---

## 第一阶段：事实抽取
先从代码中抽出结构化事实，例如：

- 入口方法 `A`
- 敏感动作 `sendSmsCode()`
- 校验动作 `checkCaptcha()`
- 条件 `if skip == true`
- 参数 `skip` 来源于 request
- 到达 sendSmsCode 的路径 P1 不经过 checkCaptcha
- 到达 sendSmsCode 的路径 P2 经过 checkCaptcha，但返回值未使用

这些事实是中间层。

---

## 第二阶段：规则判定
在事实层上写规则：

- 如果存在敏感动作，且缺少指定 guard，则报警
- 如果 guard 被调用，但 guard_result 不支配 sensitive_action，则报警
- 如果 guard 是否执行依赖用户可控参数，则报警
- 如果存在至少一条路径可达 sensitive_action 且无 guard，则报警

### 选它的理由
这样规则可以用统一语义表达，后续扩展成本低。

### 不选它的代价
规则直接绑定 AST 形态，换个代码风格就失效。

---

# 八、接口“可扩展”应该怎么实现

这是你特别强调的点，我给个架构建议。

---

## 方案：接口语义注册中心

不要只存“方法签名”，要存“接口语义”。

例如：

```yaml
guards:
  captcha_verify:
    match:
      - fqcn: com.xxx.captcha.CaptchaService.verify
      - annotation: @CaptchaCheck
    semantics:
      type: guard
      category: captcha
      result: boolean
      required_for:
        - send_sms_code
        - register
        - reset_password

  login_check:
    match:
      - fqcn: com.xxx.auth.LoginService.checkLogin
    semantics:
      type: guard
      category: auth
      required_for:
        - update_profile
        - bind_card

actions:
  send_sms_code:
    match:
      - fqcn: com.xxx.sms.SmsService.sendCode
    semantics:
      type: sensitive_action
      category: send_verify_code

  reset_password:
    match:
```

- method_pattern: "*reset*Password*"
```

这里面有 3 层含义：
1. **怎么匹配**
2. **它在业务上是什么**
3. **它和哪些动作有约束关系**

### 好处
- 新增接口时只补配置
- 同一规则可跨多个项目复用
- 业务方可以自己维护一部分映射

### 风险 / 瓶颈
- 注册中心本身会成为维护热点
- 命名漂移、封装层过多会导致识别不准
- 如果只靠方法名匹配，误报会高

### 解决建议
采用多种匹配信号：
- 全限定类名
- 方法签名
- 注解
- 包路径
- 返回值/参数特征
- 调用上下文模式

# 九、最关键的问题：如何判断“是否会被绕过”

这是最难的部分，我给你一个可落地判断模型。

---

## 定义：Guard 是否有效

对于一个敏感动作 `S`，一个 guard `G` 只有在满足下面条件时才算有效：

1. `G` 在到达 `S` 的路径上执行
2. `G` 的结果真正影响是否能执行 `S`
3. 所有可达 `S` 的路径都被 `G` 覆盖，或者被等价 guard 覆盖
4. `G` 的执行条件本身不可被低信任输入随意关闭
5. `G` 校验的对象与 `S` 操作的对象一致

也就是说，不是“调用了 check 就算安全”。

---

## 你可以用“支配关系”来建模
简化理解：

> 如果一个校验节点不能支配敏感操作节点，那就可能存在绕过路径。

例如 CFG 上：

```text
entry
 ├─ if skip=true ───────────────→ sensitiveAction
 └─ verifyCaptcha() → if ok ───→ sensitiveAction
```

这里 `verifyCaptcha()` 显然不支配 `sensitiveAction`，因为上面的分支可以绕过它。

这是很适合架构化复用的一个通用判断。

# 十一、单点故障和系统瓶颈

从架构角度，我要提醒几个风险点。

---

## 1. 单点：调用图/数据流引擎
这是整个 skill 的基础设施，如果这里能力弱，后面全是空中楼阁。

### 风险
- 跨函数分析不准
- 动态分发识别不到
- 框架代理、反射、AOP 打断调用链

### 建议
- 优先基于成熟静态分析框架做二次封装
- 不要自己从零写完整程序分析器，成本太大

---

## 2. 单点：接口语义配置中心
如果所有“关键接口识别”都靠人工维护，这里会成为扩展瓶颈。

### 风险
- 配置滞后
- 不同业务线命名不统一
- 新接口漏登记

### 建议
- 支持人工配置 + 自动候选发现
- 自动推荐相似接口归类
- 配置版本化

---

## 3. 瓶颈：路径爆炸
一做分支分析和跨过程路径追踪，复杂项目很容易爆。

### 风险
- 性能不可控
- 分析时长过长
- CI/CD 无法接入

### 建议
采用分层优化：
- 先做方法内快速筛选
- 再对命中候选做跨过程深度分析
- 对规则只追踪和敏感动作相关的路径
- 做摘要缓存（method summary）

---

## 4. 瓶颈：误报控制
业务类风险最怕误报，因为它不像 SQL 注入那样有强模式。

### 风险
- 规则一多，开发不信
- 平台上线后没人看结果

### 建议
输出必须带：
- 关键参数
- 关键调用
- 缺失的 guard
- 可绕过路径
- 置信度

最好支持：
- high：明确缺失/明确绕过
- medium：疑似弱约束
- low：需要人工确认

# 十二、技术路线建议

如果你现在要做 MVP，我建议技术上走“成熟方案 + 语义增强”，不要纯大模型也不要纯 regex。

---

## 推荐路线
### 基础层
- AST / CFG / 调用图 / 基本数据流：用成熟静态分析能力
- 不建议纯自研从零开始

### 语义层
- 你自己做“业务语义注册中心”
- 你自己做“Guard / Action / Condition”的抽象

### 规则层
- 用 DSL 或配置化规则
- 规则尽量基于中间事实，而不是直接写 AST 代码

### 可选增强
- 用 LLM 做接口语义归类、结果解释、相似绕过模式推荐
- 但**不要把核心判定交给 LLM**

---

## 为什么选这个路线
- 静态分析负责“准”
- 规则引擎负责“稳”
- 语义层负责“扩展”
- LLM 负责“增强而非裁决”

## 不选它的代价
如果你纯靠 LLM：
- 路径证据不稳定
- 结果不可重复
- 很难做 CI 集成和规则回归

如果你纯靠硬编码规则：
- 对业务语义变化不适应
- 扩展慢
- 维护成本爆炸

# 十三、建议的 MVP 范围

不要一开始就“分析所有业务风险”，会失控。  
建议先只做 3 类高价值风险：

---

## MVP-1：敏感操作缺少必要校验
例子：
- 发验证码前缺少图形验证码
- 修改手机号前缺少短信验证码
- 提现前缺少支付密码校验

---

## MVP-2：校验结果未绑定 / 可绕过
例子：
- verify 调用了但返回值未使用
- 某分支直接执行敏感动作
- `skipCheck` 来自请求参数

---

## MVP-3：关键接口未调用
例子：
- 敏感动作前没调风控
- 没调权限校验
- 没调状态校验

这 3 类是最容易抽象成统一框架的。

### 话题
设计一个用于识别业务逻辑风险的 skill，并收敛其能力边界、可扩展检测框架与可落地实现路径。

### 达成共识
- 该 skill 的核心目标可统一抽象为：**识别业务敏感操作是否具备正确、完整、不可绕过的前置约束**。
- 需求中的 5 类能力（参数分析、数据流追踪、指定接口识别、条件分支判断、绕过检测）不应割裂实现，而应作为统一风险分析框架中的基础能力与规则能力。
- 从设计方向上，采用**分层架构 + 业务语义抽象 + 规则可扩展**是合理方向。
- 需要抽象统一对象模型，至少包括：**入口点（EntryPoint）、敏感操作（SensitiveAction）、前置校验/守卫（Guard/Check）、条件（Condition）、证据路径（EvidencePath）**。

  - 校验是否在路径上执行
  - 校验结果是否真正约束了敏感操作
  - 是否覆盖所有可达路径
  - 校验执行条件是否可被低信任输入关闭
  - 校验对象与实际操作对象是否一致

  - 触发参数
  - 数据流/调用路径
  - 命中的关键接口
  - 缺失的校验或可绕过的条件

  ## 讨论结论

### 话题
设计一个用于识别业务逻辑风险的 skill，并收敛其能力边界、可扩展检测框架与可落地实现路径。

### 达成共识
- 该 skill 的核心目标可统一抽象为：**识别业务敏感操作是否具备正确、完整、不可绕过的前置约束**。
- 需求中的 5 类能力（参数分析、数据流追踪、指定接口识别、条件分支判断、绕过检测）不应割裂实现，而应作为统一风险分析框架中的基础能力与规则能力。
- 从设计方向上，采用**分层架构 + 业务语义抽象 + 规则可扩展**是合理方向。
- 需要抽象统一对象模型，至少包括：**入口点（EntryPoint）、敏感操作（SensitiveAction）、前置校验/守卫（Guard/Check）、条件（Condition）、证据路径（EvidencePath）**。
- “指定接口识别需要可扩展”是框架设计重点，不能把关键接口识别硬编码到单条规则里。
- 绕过类风险不能仅判断“是否调用了校验接口”，还需关注：
  - 校验是否在路径上执行
  - 校验结果是否真正约束了敏感操作
  - 是否覆盖所有可达路径
  - 校验执行条件是否可被低信任输入关闭
  - 校验对象与实际操作对象是否一致
- 输出结果不应只是风险列表，还应包含**证据链**，至少包括：
  - 触发参数
  - 数据流/调用路径
  - 命中的关键接口
  - 缺失的校验或可绕过的条件
  - 风险置信度
