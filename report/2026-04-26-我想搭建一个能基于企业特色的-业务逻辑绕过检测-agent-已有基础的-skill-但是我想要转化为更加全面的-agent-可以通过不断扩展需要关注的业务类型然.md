## 讨论结论

### 话题
设计一个用于接口离线评估的可扩展 Agent：先识别接口所属业务域，再复用统一的“利用方式”抽象与条件树规则引擎，发现业务逻辑风险。

### 核心分歧与演进
Architect 先提出了一个清晰分层的方案：把“业务域识别”和“风险判定”解耦，形成“标准化输入 → 业务域路由 → 利用方式抽象 → 条件树求值 → 结果落库”的主链路，并主张以业务 Plugin + 轻量 AST 规则引擎代替重型规则平台。

讨论随即集中到两个关键张力上：

1. **抽象是否值得做**
   - Architect 认为“业务异构、利用方式共性”是系统可扩展的根本，因此必须把营销、短信等具体业务统一映射到 `Bypass / Replay / Abuse / Leak` 等原语，否则 3 年后会退化为脚本堆。
   - 对这一点的质疑主要落在“映射会不会脆化、规则树会不会爆炸”。Architect 没有否认风险，而是把方案从“全局通用规则”修正为“先按 Domain + Primitive 做一级索引，再在小范围内执行条件树”，即用物理分组降低抽象失真和性能成本。

2. **规则引擎该多重**
   - Architect 先否定了 Drools 这类重型引擎，理由是离线批量场景更需要可序列化、可热更新、低运行时开销的规则执行。
   - 在性能风险被提出后，他进一步补充：不依赖复杂的规则共享算法，而通过“领域路由 + 原语分组”把规则集压缩到每次只评估 5–10 条，从而接受 O(N) 求值器的简单性，换取维护成本更低的实现。

3. **业务识别层如何演进**
   - 初始方案是正则/字典识别。
   - 在考虑同义词、命名漂移后，架构演进为“正则优先，未命中再用轻量文本分类模型”的混合路由。也就是说，识别层从纯规则被修正为“规则保可解释性，模型补覆盖率”。

整体上，讨论从“搭一个能不断扩展的业务逻辑检测 agent”逐步收敛为：**以 Plugin 为扩展单元、以 Domain + Primitive 为路由索引、以轻量 AST 为规则承载、以证据链结果模型为输出核心** 的离线评估框架。

### 达成共识
- 采用**分层解耦架构**：标准化输入、业务域识别、通用利用方式抽象、条件树规则执行、结果聚合分别独立。
- 把**业务域**与**利用方式原语**分离：例如营销和短信是不同域，但都可能映射到 `BYPASS / ABUSE / REPLAY` 等统一风险动作。
- 规则执行层优先采用**轻量自研 AST/条件树引擎**，而不是引入重型规则平台。
- 扩展机制以**业务 Plugin** 为核心，每个 Plugin 至少包含：领域识别规则、原语映射规则、条件树/规则包、风险说明与修复建议模板。
- 初期只做**离线评估**，输入来自 API 元数据、样本流量或 HAR/Swagger；不把在线阻断纳入 MVP。
- 为控制复杂度，规则执行必须按 **Domain + Primitive** 先分桶，再做局部规则判定，避免全量规则遍历。
- 结果模型必须包含**证据链**，不仅输出风险标签，还要说明命中了哪些条件、置信度多少、建议怎么修。

### 仍有分歧
| 分歧点 | 正方立场 | 反方立场 | 各自依据 | 主持人倾向 |
|--------|----------|----------|----------|-----------|
| 利用方式是否足够稳定，能否长期抽象为少数原语 | 可以，先以 `Bypass/Replay/Abuse/Leak` 建模，未来再增补 | 不一定，复杂企业业务会出现跨域欺诈、链式条件利用，固定原语可能不够 | 正方依据是复用性和可扩展性；反方依据是高阶业务攻击往往跨接口、跨状态 | 倾向正方，但要求保留原语扩展能力，避免写死枚举 |
| 业务域识别是否会成为系统瓶颈 | 可先用规则字典，后续混合小模型补足覆盖 | 业务命名漂移、语义隐藏时，识别误差会显著影响后续判定 | 正方依据是可解释、工程成本低；反方依据是误报漏报会在路由层被放大 | 倾向折中：MVP 规则优先，二期加模型兜底 |
| 条件树是否会脆化 | 会，但可通过小规则集、插件隔离和算子扩展控制 | 复杂业务条件常依赖上下文状态，静态树可能过于僵硬 | 正方依据是条件树可版本化、可审核；反方依据是业务演化速度快于规则维护速度 | 倾向正方，但把“跨接口/状态型规则”列为后续增强项 |
| 离线评估是否足以发现业务逻辑风险 | 对接口级逻辑风险筛查已足够，是高性价比起点 | 对重放、链式滥用、时序约束类问题，离线样本可能看不到真实风险 | 正方依据是离线成本低、覆盖广；反方依据是缺失运行态和多请求关联信息 | 倾向折中：先离线评估，后补多接口关联与回放验证能力 |

### 决策建议
推荐采用 **“Plugin 化业务识别 + 统一利用原语 + 轻量 AST 规则引擎”** 的分层 Agent 架构，并按 **MVP → 动态化 → 关联分析** 三阶段推进。

最有力的论据来自 Architect 的两点判断：

1. **如果不做抽象，系统会退化成脚本集合，无法持续扩展。**  
   这是该议题最根本的设计约束。用户目标不是一次性检测短信或营销，而是“能不断扩展业务类型并复用相同业务逻辑判断”。因此必须把“业务识别”与“利用方式”拆开。

2. **在离线批量评估场景中，重型规则引擎不如轻量 AST + 路由分组更合适。**  
   通过 `Domain + Primitive` 预分桶，每次只评估少量相关规则，就能用简单执行器换取更低维护成本、更好的序列化和热更新能力，这比一开始引入复杂引擎更符合工程现实。

因此最终方案应是：
- 架构上坚持分层解耦；
- 扩展上坚持 Plugin 单元；
- 规则上坚持 JSON/AST 可配置；
- 实施上坚持先做短信/营销双域 MVP；
- 风险上承认离线评估不是终局，预留多接口/关联分析能力。

## 最终推荐方案

### 一、问题抽象
把目标系统抽象成一句话：

> **对离线接口样本做“业务语义识别 + 通用风险动作映射 + 条件树规则判定”，以发现业务逻辑绕过、重放、滥用等风险。**

它不是传统漏洞扫描器，也不是纯规则匹配器，而是一个面向企业业务特征的**离线风险评估 Agent**。

核心抽象分为两层：
1. **业务域层**：接口是不是营销、短信、账户、支付等。
2. **利用方式层**：不管属于什么业务，它是否具备绕过、重放、滥用、越权泄露等可利用特征。

这两个抽象一旦成立，就能把“新增业务类型”变成“新增 Plugin”，而不是重写整个检测链路。

---

### 二、核心设计原则
1. **识别与判定解耦**  
   先判断接口属于什么业务，再判断它可能被如何利用。

2. **业务异构，利用方式收敛**  
   营销和短信规则不同，但都可以落到少数风险原语上。

3. **规则必须可配置、可版本化、可热更新**  
   业务逻辑变化快，不能把检测逻辑硬编码在程序里。

4. **结果必须可解释**  
   每个命中都要输出证据链，而不是只有一个“高危”标签。

5. **MVP 先解决 60 分问题，再考虑 90 分的泛化**  
   先支持短信/营销两类业务和单接口评估，不一开始就追求跨接口全链路推理。

### 三、推荐系统架构

#### 1. 总体分层
```text
输入层
  └─ HAR / Swagger / 样本流量 / API Meta

标准化层
  └─ 统一输出 InterfaceContext

特征提取层
  └─ path / method / params / headers / response / 文本特征 / 结构特征

业务域识别层
  └─ Domain Router
     ├─ 规则字典识别
     └─ 小模型兜底（后续）

利用方式抽象层
  └─ Primitive Mapper
     ├─ BYPASS
     ├─ REPLAY
     ├─ ABUSE
     └─ LEAK（预留）

规则执行层
  └─ AST / 条件树引擎
     ├─ 按 Domain + Primitive 分桶
     └─ 执行 Plugin 内规则

结果聚合层
  └─ EvaluationResult + evidence + confidence + remediation

插件层
  └─ SMS Plugin / Marketing Plugin / Future Plugins
```

#### 2. 模块边界
- **标准化层**：不关心风险，只负责把各种输入变成统一上下文。
- **业务域识别层**：只回答“是什么业务”。
- **利用方式抽象层**：只回答“可能怎么被利用”。
- **规则执行层**：只回答“是否满足风险条件”。
- **插件层**：承载业务差异，避免核心引擎污染。

### 五、Agent 工作流

#### Step 1：输入接入
支持以下任一输入：
- HAR 包
- Swagger/OpenAPI
- 样本接口调用日志
- 录制的请求响应对

#### Step 2：标准化
转成统一 `InterfaceContext`。

#### Step 3：业务域识别
优先使用规则字典：
- path 关键词
- 参数名
- body 字段
- response 语义

例如：
- `sms/send`、`mobile`、`template_id` → `SMS`
- `promotion`、`coupon`、`discount` → `MARKETING`

二期再加入小模型补同义词和弱显式语义。

#### Step 4：利用方式映射
根据业务域和特征，映射到原语：
- `SMS` → `ABUSE / REPLAY`
- `MARKETING` → `BYPASS / ABUSE`

#### Step 5：规则分发
按 `Domain + Primitive` 定位规则包，避免全量规则扫描。

#### Step 6：条件树执行
使用轻量 AST 求值器，对上下文事实做判断。

#### Step 7：结果输出
输出风险列表、证据链、置信度、修复建议。

### 六、业务扩展示例

#### 1. 短信业务 `SMS Plugin`

**领域识别特征**
- path 含 `sms/send`、`message/send`
- 参数含 `phone/mobile/template_id/code/type`

**典型原语**
- `ABUSE`：短信轰炸、批量滥发
- `REPLAY`：重复发送验证码
- `BYPASS`：跳过图形验证码或频率限制

**示例规则**
```json
{
  "rule_id": "sms_abuse_001",
  "domain": "SMS",
  "primitive": "ABUSE",
  "when": {
    "logic": "AND",
    "children": [
      {"fact": "path", "op": "contains", "value": "send"},
      {"fact": "body.mobile", "op": "exists"},
      {"fact": "body.template_id", "op": "exists"}
    ]
  },
  "remediation": "增加手机号、IP、设备维度频控与业务场景校验"
}
```

#### 2. 营销业务 `MARKETING Plugin`

**领域识别特征**
- path 含 `promotion/coupon/discount/claim`
- 参数含 `coupon_id/activity_id/discount`

**典型原语**
- `BYPASS`：领取条件绕过
- `ABUSE`：重复领券、滥用活动
- `REPLAY`：重复提交营销操作

**示例规则**
```json
{
  "rule_id": "marketing_abuse_001",
  "domain": "MARKETING",
  "primitive": "ABUSE",
  "when": {
    "logic": "AND",
    "children": [
      {"fact": "path", "op": "contains", "value": "coupon"},
      {"fact": "body.user_id", "op": "exists"},
      {"fact": "response.data.status", "op": "equals", "value": "success"}
    ]
  },
  "remediation": "增加用户维度幂等控制、活动次数限制和资格校验"
}
```

### 七、MVP 实施路径

#### Phase 1：最小可行版本
目标：4–6 周内跑通闭环。

**范围**
- 只支持两类业务域：`SMS`、`MARKETING`
- 输入只接一种：优先 HAR 或样本流量
- 业务识别只用规则字典
- 原语先支持：`BYPASS / REPLAY / ABUSE`
- AST 只支持：
  - `AND`
  - `OR`
  - `exists`
  - `contains`
  - `equals`

**交付物**
- 统一数据模型
- 规则执行器
- 两个 Plugin
- 基本结果输出

#### Phase 2：动态化和可维护性增强
目标：让系统真正具备扩展能力。

**增强项**
- 规则 JSON 配置化
- Plugin 注册/卸载机制
- 热更新
- 规则版本管理
- 识别层加入小模型兜底

#### Phase 3：高阶风险能力
目标：从“单接口判定”进化到“多接口业务逻辑分析”。

**增强项**
- 跨接口关联
- 回放验证
- 时序/状态判断
- 高阶原语，如 `FRAUD`

---

### 八、主要折中建议

#### 折中 1：先轻量，不上重型规则引擎
原因：当前场景是离线评估，不是复杂在线推理。轻量 AST 足够支撑 MVP 和二期。

#### 折中 2：先规则识别，不急于上模型
原因：短信/营销的显式语义强，规则更便宜、更可解释。

#### 折中 3：先做单接口分析，不强求跨接口闭环
原因：跨接口推理会显著提高建模复杂度，应放到三期。

#### 折中 4：原语先少量定义，但必须可扩展
原因：抽象太多会失控，抽象太少会不够用。先收敛到 3–4 个，再随着业务增加。

---

### 九、结论性建议
最终推荐你采用以下路线：

> **以业务 Plugin 为扩展单元，以 Domain + Primitive 为核心路由键，以轻量 AST 为规则执行内核，先落地短信/营销双域的离线评估 MVP，再逐步增强识别精度、规则动态化和跨接口关联能力。**

这是当前讨论中**扩展性、工程可行性、长期维护成本**三者之间最平衡的方案。

### 二、 分层架构与核心对象模型

#### 1. 分层架构设计
- **L1 标准化层：** 消费多源数据（Har包、Swagger、流量镜像），统一输出 `InterfaceContext`。
- **L2 业务域识别层：** 
  - **边界：** 只负责回答“这是不是营销域？是不是短信域？”，不回答“有没有漏洞”。
  - **实现：** 基于接口路径、参数名、响应特征的正则字典 + 轻量级文本分类模型。
- **L3 通用利用方式抽象层：**
  - **边界：** 将具体的业务逻辑转化为统一的攻击原语（Bypass绕过、Replay重放、Abuse滥用、Leak越权泄露）。
  - **核心：** 营销域的“篡改优惠金额”和短信域的“伪造发送者”在L2是不同的，但在L3都映射为 `Bypass` 原语。
- **L4 条件树/规则引擎层：**
  - **边界：** 接收 L3 的原语和 L2 的领域标签，执行具体的逻辑判定。
  - **机制：** 采用 AST（抽象语法树）表达条件，叶子节点为具体的数据提取器或断言。
- **L5 规则包/插件扩展机制：**
  - 基于 SPI 机制，每个业务域（如 SMS、Marketing）作为一个独立 Plugin，包含：领域识别字典 + 利用方式映射配置 + 条件树 JSON。

#### 2. 关键数据结构 (核心对象模型)
```json

### 三、 规则执行流程与扩展点设计

#### 1. 规则执行流程
1. **Ingestion：** 接口数据流入，提取 URL、Headers、Body 等特征。
2. **Routing：** Domain Router 并行运行各 Plugin 的识别器，为接口打上领域标签（如 `SMS`）。
3. **Mapping：** 根据 Domain 标签，加载对应的 Exploit Primitive 映射规则（如 `SMS` 域映射出 `ABUSE` 短信轰炸风险）。
4. **Evaluation：** Condition Tree Engine 获取该 Domain + Primitive 下的所有条件树，以 `Interface Context` 为事实基座进行求值。
5. **Output：** 命中的叶子节点组装成证据链，与风险类型一起落库。

#### 2. 扩展点设计（三年后的演进考量）
三年后，业务域从 2 个扩到 20 个，接口量从万级到百万级，扩展点必须前置：
- **Domain 识别扩展：** 规则从正则升级为小模型，只需实现 `DomainIdentifier` 接口并注册，不影响路由层逻辑。
- **利用方式扩展：** 当前定义了 Bypass/Replay/Abuse，未来新增如 `FRAUD(欺诈)` 原语，只需在枚举中增加，映射层通过配置下推，无需改动 Engine 核心代码。
- **条件树算子扩展：** 预留 `CustomOperator` SPI，比如未来需要基于图数据库做“关系链越权”判定，可实现 `GraphPathOperator` 插入条件树。

### 四、 技术选型与复杂度权衡

#### 1. 条件树/规则引擎
- **选型：自研轻量级 AST 求值器**（而非 Drools 或 Easy-Rules）
- **核心理由：** 离线评估需要批量处理百万级接口，Drools 太重，含有大量 Rete 网络构建开销，且不利于序列化和版本控制。自研 AST 求值器可以直接用 JSON 存储，无状态，并发度极高。
- **最大风险：** 缺乏 Rete 算法的规则共享机制，如果有 1000 条规则，每条都必须遍历执行，P99 延迟可能退化为 O(N)。
- **折中方案（回应 Pragmatist 的成本考量）：** 引入 `Domain + Primitive` 作为一级索引进行规则分组，每次求值只加载 5-10 条相关规则，将 O(N) 降为 O(1) 常数级，用“物理隔离”替代“算法共享”，成本极低且保留架构优势。

#### 2. 业务域识别层
- **选型：正则字典树 + 预训练 NLP 小模型 (如 FastText) 混合路由**
- **核心理由：** 纯正则无法覆盖同义词（如 promo/coupon/discount），纯模型存在黑盒不可解释且 P99 > 2s。混合路由优先正则，未命中再降级模型。
- **最大风险：** 模型推理在离线批量任务中可能成为 CPU/Memory 瓶颈。

#### 3. 复杂度反思（直面 Challenger 的假设）
引入条件树和利用方式抽象，**最大的新复杂度在于“映射的脆化”**。如果 API 既属于营销域又属于支付域，条件树判定时产生交叉，可能会产生组合爆炸或误报。
- **值不值？** 值得。如果不做抽象，每个业务域从 0 写评估逻辑，3 年后系统将沦为“脚本集合”，无法维护。引入 AST 带来的解析开销，相比复用性的提升，是微不足道的。但必须在 Domain Router 层做**优先级断言**（互斥优先，重叠则 fork 出多个评估上下文并行跑）。

### 五、 技术演进路线

1. **Phase 1 (MVP)：** 纯正则识别 Domain，硬编码利用方式映射，AST 引擎仅支持 AND/OR/Contains 三种算子。跑通 SMS/Marketing 双域。
2. **Phase 2 (解耦与动态化)：** 引入 Plugin 机制，规则由 JSON 数据库下发，支持热更新；Domain 识别引入 FastText 模型。
3. **Phase 3 (闭环与高阶算子)：** AST 支持 Datalog 推理算子（如关联分析：A接口的返回值是B接口的越权条件），从单接口评估演进为多接口组合逻辑评估，此时系统可抗住 10万+/天的接口增量评估，P99 < 200ms。

  "path": "/api/v1/coupon/claim",
  "method": "POST",
  "param_names": ["coupon_id", "user_id", "amount"],
  "has_request_body": true,
  "response_status_codes": [200, 403]
}
```
**说明**：MVP 阶段不解析参数值是否合法、不猜字段业务含义（如 `amount` 是金额），只拿“参数名存在性”做条件树断言。这能降低 70% 的解析复杂度，但能覆盖“缺少 user_id 校验”这类基础规则。

### 四、规则配置方式：用 GitOps 控制维护成本

Architect 提到了规则包与 SPI，这在长期是对的，但 MVP 阶段维护成本最高的不是“扩展”，而是“规则腐败和没人知道规则写了什么”。

**MVP 规则配置采用 Git 管理的 YAML：**
```yaml
# rules/marketing.yaml
domain: MARKETING
rules:
  - rule_id: MKT-BYPASS-001
    primitive: BYPASS
    description: "营销领券接口缺少金额服务端校验"
    severity: HIGH
    confidence_base: 0.7
    condition:
      and:
        - contains: { fact: "path", value: "claim" }
        - eq: { fact: "method", value: "POST" }
        - exists: { fact: "params", value: "amount" }  # 前端传了amount，但后端应忽略
    exclude_paths: ["/api/v1/coupon/claim/query"]  # 白名单降噪
    owner: "security-team"
    expiry_date: "2024-12-31"
```

**为什么这样配？**
- 所有规则与代码同仓库发布，MR 走 CR，天然有版本追溯。
- 新增一个规则 = 改 YAML + 提 MR + 重启 Agent，MVP 阶段规则变更频率一周不到一次，这个成本远低于维护一套 Plugin 热加载框架。
- 每条规则带 `owner` 和 `expiry_date`，防止三个月后 YAML 膨胀到无人敢删。

### 五、分阶段实施计划（4 周日历时间）

**第 1 周：数据接入与上下文标准化**
- **周一-周二**：写 Swagger 2.0/3.0 解析器，统一输出 `InterfaceContext` JSON（方法、路径、参数名列表）。
- **周三**：写 Har 解析器（如果有存量），对齐到同一上下文结构。
- **周四**：做基线普查——拿公司核心系统的所有 Swagger 跑一遍，统计接口总量、参数命名规范度、未识别 Domain 的比例。**如果未识别率 > 30%，说明关键词库不够，MVP 范围必须缩到仅覆盖 1 个核心系统，而非全公司。**
- **周五**：样本清洗，沉淀 200 个接口作为测试集。

**第 2 周：业务路由与规则求值器**
- **周一**：实现 Trie/正则 Domain Router，YAML 配置关键词映射（sms, marketing）。
- **周二-周三**：手写 `JSON Rule Evaluator`（200 行，递归解析 AND/OR/LEAF 节点），支持运算符：`contains`, `exists`, `eq`, `not`。不要 Parser，直接 `json.Unmarshal` 后求值。
- **周四**：定义**短信域**规则集（3-5 条），例如：path 包含 send + param 包含 phone + method=POST → ABUSE 风险。
- **周五**：跑通一个短信接口的端到端流程（Swagger 输入 → 识别 SMS → 命中规则 → 输出 JSON 报告）。

**第 3 周：营销域规则与评估闭环**
- **周一-周二**：定义**营销域**规则集（3-5 条），例如：path 包含 claim/coupon + method=POST + param 存在 amount → BYPASS 风险。
- **周三**：实现评估结果模型，设计置信度打分策略（MVP 简单按命中叶子节点数 / 总风险特征数 计算，不引入模型）。
- **周四**：批处理 Runner（支持一次性扫描 1000 个接口，产出聚合报告）。
- **周五**：人工复核——拉安全工程师看报告，标记真阳性/误报，调整阈值。

**第 4 周：降噪、集成与交付**
- **周一-周二**：接入 CI（如 nightly build 跑扫描），加入**全局白名单**（过滤 `/ health`, `/metrics`, `/public/**`）和**置信度硬阈值**（<0.6 不入报告）。这是 Architect 方案里没提但必须加的“防洪堤”。
- **周三**：用 5 个真实项目的接口做验收，目标是**产出 < 50 条高置信度（>0.7）告警，且人工复核真阳性 ≥ 5 条**。
- **周四**：补单元测试（Eval 核心求值器覆盖率 > 80%，规则 YAML 加 Schema 校验，防止手滑写错 `and` 拼写）。
- **周五**：输出规则编写模板与 README，教会安全工程师如何新增规则，移交维护权。

### 七、对 Architect 方案的落地调整建议（非否定，可执行）

Architect 的五年架构图很清晰，但直接按 Phase 1 落地仍有几处对当前团队而言**成本前置、收益后置**。建议做以下三处调整：

1. **MVP 规则引擎从“自研 AST”降级为“JSON 求值器”**
   - **原因**：AST 的序列化与无状态优势在规则量 < 50 条时无法体现，反而需要定义 Node 接口、Visitor 模式等，开发成本约 +1 人周。
   - **调整**：MVP 用 JSON 数组描述条件树，手写递归求值器。代码量 200 行，零外部依赖。在 Phase 2 规则量破 200 条或需要可视化编辑器时，再迁入 AST。

2. **MVP 放弃 FastText，Trie 树 + 正则足够支撑双域**
   - **原因**：短信（send, sms, message）和营销（coupon, promo, claim）的英文关键词高度集中，20 个关键词的 Trie 匹配覆盖度可达 80% 以上。引入 FastText 需要标注数据、模型训练、推理服务，至少 +2 人周，且黑盒结果不利于安全工程师解释。
   - **调整**：将 NLP 识别作为 L2 预留的扩展点（保留 `DomainIdentifier` 接口），但 MVP 实现类用 `RegexIdentifier`。Phase 2 再无缝替换。

3. **MVP 放弃 Plugin SPI，改用“配置文件 + 策略工厂”**
   - **原因**：动态 Plugin 涉及类隔离、版本兼容性、热加载监控，MVP 阶段规则变更一周一次，不需要热更新。SPI 框架会吃掉 1 人周且引入第一个外部故障点。
   - **调整**：MVP 每个 Domain 对应一个 `domain_xxx.yaml` 配置，Agent 启动时加载到内存 Map。YAML 变更随 Agent 发版走。等 Domain 数量 > 10 个后，再按 Architect 设计迁入 Plugin 机制。

[Synthesizer] ## 讨论结论

### 话题
设计一个用于接口离线评估的可扩展 Agent：先识别接口所属业务域，再复用统一的“利用方式”抽象与条件树规则引擎，发现业务逻辑风险。

### 核心分歧与演进
Architect 先提出了一个清晰分层的方案：把“业务域识别”和“风险判定”解耦，形成“标准化输入 → 业务域路由 → 利用方式抽象 → 条件树求值 → 结果落库”的主链路，并主张以业务 Plugin + 轻量 AST 规则引擎代替重型规则平台。

讨论随即集中到两个关键张力上：

1. **抽象是否值得做**
   - Architect 认为“业务异构、利用方式共性”是系统可扩展的根本，因此必须把营销、短信等具体业务统一映射到 `Bypass / Replay / Abuse / Leak` 等原语，否则 3 年后会退化为脚本堆。
   - 对这一点的质疑主要落在“映射会不会脆化、规则树会不会爆炸”。Architect 没有否认风险，而是把方案从“全局通用规则”修正为“先按 Domain + Primitive 做一级索引，再在小范围内执行条件树”，即用物理分组降低抽象失真和性能成本。

2. **规则引擎该多重**
   - Architect 先否定了 Drools 这类重型引擎，理由是离线批量场景更需要可序列化、可热更新、低运行时开销的规则执行。
   - 在性能风险被提出后，他进一步补充：不依赖复杂的规则共享算法，而通过“领域路由 + 原语分组”把规则集压缩到每次只评估 5–10 条，从而接受 O(N) 求值器的简单性，换取维护成本更低的实现。

3. **业务识别层如何演进**
   - 初始方案是正则/字典识别。
   - 在考虑同义词、命名漂移后，架构演进为“正则优先，未命中再用轻量文本分类模型”的混合路由。也就是说，识别层从纯规则被修正为“规则保可解释性，模型补覆盖率”。

整体上，讨论从“搭一个能不断扩展的业务逻辑检测 agent”逐步收敛为：**以 Plugin 为扩展单元、以 Domain + Primitive 为路由索引、以轻量 AST 为规则承载、以证据链结果模型为输出核心** 的离线评估框架。

### 达成共识
- 采用**分层解耦架构**：标准化输入、业务域识别、通用利用方式抽象、条件树规则执行、结果聚合分别独立。
- 把**业务域**与**利用方式原语**分离：例如营销和短信是不同域，但都可能映射到 `BYPASS / ABUSE / REPLAY` 等统一风险动作。
- 规则执行层优先采用**轻量自研 AST/条件树引擎**，而不是引入重型规则平台。
- 扩展机制以**业务 Plugin** 为核心，每个 Plugin 至少包含：领域识别规则、原语映射规则、条件树/规则包、风险说明与修复建议模板。
- 初期只做**离线评估**，输入来自 API 元数据、样本流量或 HAR/Swagger；不把在线阻断纳入 MVP。
- 为控制复杂度，规则执行必须按 **Domain + Primitive** 先分桶，再做局部规则判定，避免全量规则遍历。
- 结果模型必须包含**证据链**，不仅输出风险标签，还要说明命中了哪些条件、置信度多少、建议怎么修。

### 仍有分歧
| 分歧点 | 正方立场 | 反方立场 | 各自依据 | 主持人倾向 |
|--------|----------|----------|----------|-----------|
| 利用方式是否足够稳定，能否长期抽象为少数原语 | 可以，先以 `Bypass/Replay/Abuse/Leak` 建模，未来再增补 | 不一定，复杂企业业务会出现跨域欺诈、链式条件利用，固定原语可能不够 | 正方依据是复用性和可扩展性；反方依据是高阶业务攻击往往跨接口、跨状态 | 倾向正方，但要求保留原语扩展能力，避免写死枚举 |
| 业务域识别是否会成为系统瓶颈 | 可先用规则字典，后续混合小模型补足覆盖 | 业务命名漂移、语义隐藏时，识别误差会显著影响后续判定 | 正方依据是可解释、工程成本低；反方依据是误报漏报会在路由层被放大 | 倾向折中：MVP 规则优先，二期加模型兜底 |
| 条件树是否会脆化 | 会，但可通过小规则集、插件隔离和算子扩展控制 | 复杂业务条件常依赖上下文状态，静态树可能过于僵硬 | 正方依据是条件树可版本化、可审核；反方依据是业务演化速度快于规则维护速度 | 倾向正方，但把“跨接口/状态型规则”列为后续增强项 |
| 离线评估是否足以发现业务逻辑风险 | 对接口级逻辑风险筛查已足够，是高性价比起点 | 对重放、链式滥用、时序约束类问题，离线样本可能看不到真实风险 | 正方依据是离线成本低、覆盖广；反方依据是缺失运行态和多请求关联信息 | 倾向折中：先离线评估，后补多接口关联与回放验证能力 |

### 决策建议
推荐采用 **“Plugin 化业务识别 + 统一利用原语 + 轻量 AST 规则引擎”** 的分层 Agent 架构，并按 **MVP → 动态化 → 关联分析** 三阶段推进。

最有力的论据来自 Architect 的两点判断：

1. **如果不做抽象，系统会退化成脚本集合，无法持续扩展。**  
   这是该议题最根本的设计约束。用户目标不是一次性检测短信或营销，而是“能不断扩展业务类型并复用相同业务逻辑判断”。因此必须把“业务识别”与“利用方式”拆开。

2. **在离线批量评估场景中，重型规则引擎不如轻量 AST + 路由分组更合适。**  
   通过 `Domain + Primitive` 预分桶，每次只评估少量相关规则，就能用简单执行器换取更低维护成本、更好的序列化和热更新能力，这比一开始引入复杂引擎更符合工程现实。

因此最终方案应是：
- 架构上坚持分层解耦；
- 扩展上坚持 Plugin 单元；
- 规则上坚持 JSON/AST 可配置；
- 实施上坚持先做短信/营销双域 MVP；
- 风险上承认离线评估不是终局，预留多接口/关联分析能力。

## 最终推荐方案

### 一、问题抽象
把目标系统抽象成一句话：

> **对离线接口样本做“业务语义识别 + 通用风险动作映射 + 条件树规则判定”，以发现业务逻辑绕过、重放、滥用等风险。**

它不是传统漏洞扫描器，也不是纯规则匹配器，而是一个面向企业业务特征的**离线风险评估 Agent**。

核心抽象分为两层：
1. **业务域层**：接口是不是营销、短信、账户、支付等。
2. **利用方式层**：不管属于什么业务，它是否具备绕过、重放、滥用、越权泄露等可利用特征。

这两个抽象一旦成立，就能把“新增业务类型”变成“新增 Plugin”，而不是重写整个检测链路。

---

### 二、核心设计原则
1. **识别与判定解耦**  
   先判断接口属于什么业务，再判断它可能被如何利用。

2. **业务异构，利用方式收敛**  
   营销和短信规则不同，但都可以落到少数风险原语上。

3. **规则必须可配置、可版本化、可热更新**  
   业务逻辑变化快，不能把检测逻辑硬编码在程序里。

4. **结果必须可解释**  
   每个命中都要输出证据链，而不是只有一个“高危”标签。

5. **MVP 先解决 60 分问题，再考虑 90 分的泛化**  
   先支持短信/营销两类业务和单接口评估，不一开始就追求跨接口全链路推理。

---

### 三、推荐系统架构

#### 1. 总体分层
```text
输入层
  └─ HAR / Swagger / 样本流量 / API Meta

标准化层
  └─ 统一输出 InterfaceContext

特征提取层
  └─ path / method / params / headers / response / 文本特征 / 结构特征

业务域识别层
  └─ Domain Router
     ├─ 规则字典识别
     └─ 小模型兜底（后续）

利用方式抽象层
  └─ Primitive Mapper
     ├─ BYPASS
     ├─ REPLAY
     ├─ ABUSE
     └─ LEAK（预留）

规则执行层
  └─ AST / 条件树引擎
     ├─ 按 Domain + Primitive 分桶
     └─ 执行 Plugin 内规则

结果聚合层
  └─ EvaluationResult + evidence + confidence + remediation

插件层
  └─ SMS Plugin / Marketing Plugin / Future Plugins
```

#### 2. 模块边界
- **标准化层**：不关心风险，只负责把各种输入变成统一上下文。
- **业务域识别层**：只回答“是什么业务”。
- **利用方式抽象层**：只回答“可能怎么被利用”。
- **规则执行层**：只回答“是否满足风险条件”。
- **插件层**：承载业务差异，避免核心引擎污染。

### 五、Agent 工作流

#### Step 1：输入接入
支持以下任一输入：
- HAR 包
- Swagger/OpenAPI
- 样本接口调用日志
- 录制的请求响应对

#### Step 2：标准化
转成统一 `InterfaceContext`。

#### Step 3：业务域识别
优先使用规则字典：
- path 关键词
- 参数名
- body 字段
- response 语义

例如：
- `sms/send`、`mobile`、`template_id` → `SMS`
- `promotion`、`coupon`、`discount` → `MARKETING`

二期再加入小模型补同义词和弱显式语义。

#### Step 4：利用方式映射
根据业务域和特征，映射到原语：
- `SMS` → `ABUSE / REPLAY`
- `MARKETING` → `BYPASS / ABUSE`

#### Step 5：规则分发
按 `Domain + Primitive` 定位规则包，避免全量规则扫描。

#### Step 6：条件树执行
使用轻量 AST 求值器，对上下文事实做判断。

#### Step 7：结果输出
输出风险列表、证据链、置信度、修复建议。

### 六、业务扩展示例

#### 1. 短信业务 `SMS Plugin`

**领域识别特征**
- path 含 `sms/send`、`message/send`
- 参数含 `phone/mobile/template_id/code/type`

**典型原语**
- `ABUSE`：短信轰炸、批量滥发
- `REPLAY`：重复发送验证码
- `BYPASS`：跳过图形验证码或频率限制

**示例规则**
```json
{
  "rule_id": "sms_abuse_001",
  "domain": "SMS",
  "primitive": "ABUSE",
  "when": {
    "logic": "AND",
    "children": [
      {"fact": "path", "op": "contains", "value": "send"},
      {"fact": "body.mobile", "op": "exists"},
      {"fact": "body.template_id", "op": "exists"}
    ]
  },
  "remediation": "增加手机号、IP、设备维度频控与业务场景校验"
}
```

#### 2. 营销业务 `MARKETING Plugin`

**领域识别特征**
- path 含 `promotion/coupon/discount/claim`
- 参数含 `coupon_id/activity_id/discount`

**典型原语**
- `BYPASS`：领取条件绕过
- `ABUSE`：重复领券、滥用活动
- `REPLAY`：重复提交营销操作

**示例规则**
```json
{
  "rule_id": "marketing_abuse_001",
  "domain": "MARKETING",
  "primitive": "ABUSE",
  "when": {
    "logic": "AND",
    "children": [
      {"fact": "path", "op": "contains", "value": "coupon"},
      {"fact": "body.user_id", "op": "exists"},
      {"fact": "response.data.status", "op": "equals", "value": "success"}
    ]
  },
  "remediation": "增加用户维度幂等控制、活动次数限制和资格校验"
}
```

### 七、MVP 实施路径

#### Phase 1：最小可行版本
目标：4–6 周内跑通闭环。

**范围**
- 只支持两类业务域：`SMS`、`MARKETING`
- 输入只接一种：优先 HAR 或样本流量
- 业务识别只用规则字典
- 原语先支持：`BYPASS / REPLAY / ABUSE`
- AST 只支持：
  - `AND`
  - `OR`
  - `exists`
  - `contains`
  - `equals`

**交付物**
- 统一数据模型
- 规则执行器
- 两个 Plugin
- 基本结果输出

#### Phase 2：动态化和可维护性增强
目标：让系统真正具备扩展能力。

**增强项**
- 规则 JSON 配置化
- Plugin 注册/卸载机制
- 热更新
- 规则版本管理
- 识别层加入小模型兜底

#### Phase 3：高阶风险能力
目标：从“单接口判定”进化到“多接口业务逻辑分析”。

**增强项**
- 跨接口关联
- 回放验证
- 时序/状态判断
- 高阶原语，如 `FRAUD`

### 八、主要折中建议

#### 折中 1：先轻量，不上重型规则引擎
原因：当前场景是离线评估，不是复杂在线推理。轻量 AST 足够支撑 MVP 和二期。

#### 折中 2：先规则识别，不急于上模型
原因：短信/营销的显式语义强，规则更便宜、更可解释。

#### 折中 3：先做单接口分析，不强求跨接口闭环
原因：跨接口推理会显著提高建模复杂度，应放到三期。

#### 折中 4：原语先少量定义，但必须可扩展
原因：抽象太多会失控，抽象太少会不够用。先收敛到 3–4 个，再随着业务增加。

---

### 九、结论性建议
最终推荐你采用以下路线：

> **以业务 Plugin 为扩展单元，以 Domain + Primitive 为核心路由键，以轻量 AST 为规则执行内核，先落地短信/营销双域的离线评估 MVP，再逐步增强识别精度、规则动态化和跨接口关联能力。**

这是当前讨论中**扩展性、工程可行性、长期维护成本**三者之间最平衡的方案。
