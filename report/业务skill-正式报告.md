# 业务风险分析 Skill 方案评审正式报告

- 生成时间：2026-04-23
- 来源：`agora/report/业务skill.md` 完整讨论日志整理
- 评审主题：设计一个用于识别业务逻辑风险的 Skill，并收敛其能力边界、可扩展检测框架与可落地实现路径

## 一、评审结论

本次讨论有效收敛出了一个可推进的方向：

- 该 Skill 的核心目标应统一抽象为：识别业务敏感操作是否具备正确、完整、不可绕过的前置约束。
- 参数分析、数据流追踪、关键接口识别、条件分支判断、绕过检测，应该作为统一风险分析框架中的基础分析能力，而不是分别堆叠的独立规则。
- 从长期演进看，采用“程序分析基础层 + 业务语义层 + 规则引擎层 + 结果输出层”的分层架构是正确方向。
- 从短期落地看，在输入形态、静态分析底座、目标语言与框架尚未明确前，不应直接进入 DSL 和细粒度模块设计，应先做轻量 MVP。

总体判断：

- 长期架构方向正确。
- 短期落地路径应降级。
- 进入实现前必须先补齐关键前提信息。

## 二、已达成共识

### 1. Skill 的核心问题定义

该 Skill 不应被定义为“扫描所有业务风险”，而应聚焦于：

- 敏感操作前是否存在必要校验
- 校验是否真正约束了敏感操作
- 校验是否覆盖所有可达路径
- 校验是否可被低信任输入绕过
- 关键业务接口是否按预期被调用

### 2. 统一对象模型是必要的

为保证规则扩展性与证据链一致性，需要至少抽象以下对象：

- `EntryPoint`：业务入口
- `SensitiveAction`：敏感操作
- `Guard / Check`：前置校验
- `Condition`：控制条件
- `EvidencePath`：证据路径

### 3. 输出必须包含证据链

最终结果不应只是“命中规则”，还应包含：

- 触发参数
- 数据流/调用路径
- 命中的关键接口
- 缺失或可绕过的校验条件
- 风险置信度

### 4. MVP 范围必须收敛

第一阶段优先关注三类高价值风险：

1. 敏感操作缺少必要校验
2. 校验存在但结果未绑定 / 可绕过
3. 关键接口未调用

## 三、主要分歧

### 1. 初期实现路线

- 观点 A：按完整目标架构设计，围绕调用图、数据流、CFG、路径覆盖构建统一框架。
- 观点 B：先降级为轻量 AST / 模式匹配方案，尽快交付可用结果。

建议：先确认输入与底座能力，再决定是“完整架构 MVP”还是“轻量规则 MVP”。

### 2. 绕过检测建模深度

- 观点 A：用支配关系、路径覆盖、条件可控、校验绑定关系做统一建模。
- 观点 B：工程代价太高，初期只检测明显可疑模式。

建议：初期以可疑模式检测为主，统一模型保留为中长期演进方向。

### 3. 接口扩展机制

- 观点 A：建立接口语义注册中心。
- 观点 B：MVP 先用轻量配置表，不要引入重型中心。

建议：先轻量配置化，后续再演进为语义注册中心。

### 4. DSL 设计时机

- 观点 A：可以先定义 DSL / rule schema。
- 观点 B：输入与底座不明时，DSL 容易脱离现实。

建议：先补齐前提信息，再定稿 DSL；当前只适合保留最小草案。

## 四、决策建议

建议采用“目标架构不变、实现路径分阶段降级”的策略：

### 阶段 0：先锁定三项前提

在实现前，先明确：

1. 输入到底是什么：源码文本、AST、IR、调用图，还是已有污点分析结果
2. 团队现有能力：是否已有 SAST 引擎、语法树服务、调用图或可复用分析底座
3. 目标语言 / 框架范围：例如 Java Spring、Go、Python、Node 等

### 阶段 1：做轻量 MVP

在底座不明的情况下，优先采用：

- 轻量 AST / TreeSitter 模式匹配
- 有限的数据流与条件判断
- 轻量接口配置
- 规则命中 + 证据链输出

优先覆盖：

- 缺少前置校验
- 校验结果未绑定
- 明显条件绕过
- 关键接口缺失

### 阶段 2：能力升级

当静态分析底座具备后，再考虑引入：

- 跨过程数据流
- CFG / 支配关系
- 更强的路径覆盖
- 统一绕过建模
- 语义注册中心

## 五、行动项

- [ ] 明确输入形态与分析边界

- [ ] 梳理第一批敏感操作与 Guard/Check 清单
- [ ] 决定第一阶段走轻量规则还是接入现有底座
- [ ] 起草最小规则集与 fallback 行为约束

## 六、风险提示

- 最大风险是默认假设底层静态分析能力存在，实际却没有。
- 若过早追求通用且精确的绕过检测，会遭遇路径爆炸、别名分析、AOP/代理等工程瓶颈。
- 若一开始试图覆盖全部业务风险，范围会迅速失控。
- 若接口扩展机制设计过重，配置中心本身会成为维护瓶颈。
- 若输出缺少证据链与置信度，结果难以获得业务方信任。
